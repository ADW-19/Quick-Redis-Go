## 8.5 消息队列

### 场景故事：电商订单处理的异步需求

**故事背景**：
某电商平台在促销活动期间，订单量激增，导致系统处理缓慢。技术团队发现，订单创建后需要立即处理支付、库存扣减、物流通知等多个操作，这些同步操作导致订单响应时间过长，用户体验差。同时，系统耦合度高，一个环节失败会影响整个订单流程。

**业务需求**：
1. 提高订单处理速度，减少用户等待时间
2. 解耦系统组件，提高系统可靠性
3. 应对流量峰值，保证系统稳定
4. 支持消息重试机制，确保消息不丢失
5. 实现消息的顺序处理

### 应用场景

- 异步处理：将耗时操作异步化，提高系统响应速度
- 解耦系统：降低系统组件间的耦合度
- 流量削峰：缓冲突发流量，保护后端系统
- 事件驱动：基于事件的系统架构

### 方案对比

**方案一：基于数据库的消息队列**
- **优点**：
  - 实现简单，利用现有数据库
  - 数据持久化，可靠性高
  - 事务支持，数据一致性好
- **缺点**：
  - 性能差，数据库成为瓶颈
  - 并发能力有限
  - 难以实现复杂的消息队列功能
  - 不支持消息过期和优先级

**方案二：专业消息队列（如RabbitMQ、Kafka）**
- **优点**：
  - 功能丰富，支持多种消息模式
  - 高可靠性，支持消息持久化
  - 支持消息确认机制
  - 支持消息分区和负载均衡
  - 生态成熟，工具完善
- **缺点**：
  - 部署和维护复杂
  - 资源消耗较大
  - 学习成本较高
  - 启动时间长

**方案三：基于Redis的消息队列**
- **优点**：
  - 性能高，支持高并发
  - 实现简单，API友好
  - 内存操作，响应速度快
  - 支持阻塞操作，实现简单的消费者模式
  - 与Redis其他功能集成方便
- **缺点**：
  - 消息可能丢失（在Redis故障情况下）
  - 不支持复杂的消息队列功能
  - 消息堆积可能导致内存压力
  - 不支持消息确认机制

**选择理由**：
考虑到电商平台的订单处理需求，选择基于Redis的消息队列方案。Redis消息队列实现简单，性能高，能够满足异步处理和流量削峰的需求。虽然在可靠性方面不如专业消息队列，但对于大多数电商场景来说已经足够。如果需要更高的可靠性，可以考虑使用专业消息队列。

### 消息队列原理

**核心原理**：利用Redis的List数据结构实现消息队列

**List特性**：
- 有序性：按照插入顺序排序
- 双向操作：支持从两端操作
- 阻塞操作：支持阻塞式弹出
- 高性能：插入、删除都是O(1)

### 消息队列工作流程

```
消息队列工作流程：
+----------------+     发送消息    +----------------+     LPUSH命令   +----------------+
|   生产者       | --------------> |   应用服务器   | --------------> |   Redis服务器  |
+----------------+                 +----------------+                 +----------------+
        |                                                                   |
        |     消息入队成功                                                 |
        +<------------------                                               |
        |                                                                   |
        |                                                                   |
        |                                                                   |
+----------------+     消费消息    +----------------+     BLPOP命令   +----------------+
|   消费者       | --------------> |   应用服务器   | --------------> |   Redis服务器  |
+----------------+                 +----------------+                 +----------------+
        |                                                                   |
        |     消息出队                                                     |
        +<------------------                                               |
        |                   |                                               |
        |                   v                                               |
        |           +----------------+                                      |
        |           |   处理消息      |                                      |
        |           +----------------+                                      |
```

### 代码实现

```go
package main

import (
	"context"
	"fmt"
	"time"

	"github.com/go-redis/redis/v8"
)

// MessageQueue 消息队列
type MessageQueue struct {
	rdb  *redis.Client
	name string
}

// NewMessageQueue 创建消息队列
func NewMessageQueue(rdb *redis.Client, name string) *MessageQueue {
	return &MessageQueue{
		rdb:  rdb,
		name: name,
	}
}

// Push 发送消息
func (q *MessageQueue) Push(ctx context.Context, message string) error {
	return q.rdb.LPush(ctx, q.name, message).Err()
}

// Pop 消费消息（非阻塞）
func (q *MessageQueue) Pop(ctx context.Context) (string, error) {
	result, err := q.rdb.RPop(ctx, q.name).Result()
	if err == redis.Nil {
		return "", nil // 队列为空
	}
	return result, err
}

// BPop 消费消息（阻塞）
func (q *MessageQueue) BPop(ctx context.Context, timeout time.Duration) (string, error) {
	result, err := q.rdb.BRPop(ctx, timeout, q.name).Result()
	if err == redis.Nil {
		return "", nil // 超时
	}
	if err != nil {
		return "", err
	}
	return result[1], nil
}

// Len 获取队列长度
func (q *MessageQueue) Len(ctx context.Context) (int64, error) {
	return q.rdb.LLen(ctx, q.name).Result()
}

// Range 获取队列内容
func (q *MessageQueue) Range(ctx context.Context, start, stop int64) ([]string, error) {
	return q.rdb.LRange(ctx, q.name, start, stop).Result()
}

// Clear 清空队列
func (q *MessageQueue) Clear(ctx context.Context) error {
	return q.rdb.Del(ctx, q.name).Err()
}

// MQService 消息队列服务
type MQService struct {
	rdb *redis.Client
}

// NewMQService 创建消息队列服务
func NewMQService(rdb *redis.Client) *MQService {
	return &MQService{rdb: rdb}
}

// GetQueue 获取消息队列
func (s *MQService) GetQueue(name string) *MessageQueue {
	return NewMessageQueue(s.rdb, name)
}

// 使用示例
func main() {
	ctx := context.Background()

	// 创建Redis客户端
	rdb := redis.NewClient(&redis.Options{
		Addr: "localhost:6379",
	})

	// 创建消息队列服务
	mqService := NewMQService(rdb)

	// 获取消息队列
	queue := mqService.GetQueue("queue:orders")

	// 生产者：发送消息
	go func() {
		for i := 1; i <= 10; i++ {
			message := fmt.Sprintf("订单 #%d", i)
			err := queue.Push(ctx, message)
			if err != nil {
				fmt.Printf("发送消息失败: %v\n", err)
				continue
			}
			fmt.Printf("发送消息: %s\n", message)
			time.Sleep(500 * time.Millisecond)
		}
	}()

	// 消费者：消费消息
	go func() {
		for {
			message, err := queue.BPop(ctx, 1*time.Second)
			if err != nil {
				fmt.Printf("消费消息失败: %v\n", err)
				continue
			}
			if message == "" {
				// 超时，继续等待
				continue
			}
			fmt.Printf("消费消息: %s\n", message)
			// 模拟处理时间
			time.Sleep(1 * time.Second)
		}
	}()

	// 等待处理完成
	time.Sleep(15 * time.Second)

	// 获取队列长度
	length, err := queue.Len(ctx)
	if err != nil {
		fmt.Printf("获取队列长度失败: %v\n", err)
		return
	}
	fmt.Printf("\n队列剩余消息数: %d\n", length)
}
```

### 消息队列的高级功能

#### 延迟消息队列
- **实现方式**：使用Sorted Set，分数为过期时间戳
- **应用场景**：定时任务、延迟通知

#### 优先级队列
- **实现方式**：使用多个List，不同优先级对应不同List
- **应用场景**：紧急任务处理

#### 消息重试机制
- **实现方式**：失败消息放入重试队列
- **应用场景**：保证消息可靠性

#### 消息确认机制
- **实现方式**：使用ACK机制
- **应用场景**：确保消息被正确处理

### 消息队列的性能优化

1. **批量操作**：使用Pipeline批量发送消息
2. **合理设置超时**：避免消费者长时间阻塞
3. **监控队列长度**：及时发现消息积压
4. **使用Redis集群**：提高并发处理能力

### 最佳实践

1. **队列命名**：使用业务前缀+功能描述
2. **消息格式**：使用JSON格式，便于解析
3. **错误处理**：妥善处理发送和消费过程中的错误
4. **监控告警**：设置队列长度和处理时间的监控
5. **数据备份**：定期备份队列数据
