## 8.1 缓存设计

### 场景故事：电商网站的性能挑战

**故事背景**：
某电商网站在促销活动期间，访问量激增，系统响应缓慢，用户投诉不断。技术团队发现主要问题是数据库压力过大，尤其是商品详情页的查询请求，每个用户访问都会触发多次数据库查询，导致数据库服务器CPU使用率飙升到90%以上。

**业务需求**：
1. 提高商品详情页的响应速度，目标是页面加载时间控制在1秒以内
2. 减轻数据库压力，避免数据库成为性能瓶颈
3. 确保数据的实时性，商品价格和库存信息需要及时更新
4. 系统需要支持高并发访问，应对促销期间的流量峰值

### 应用场景

- 热点数据缓存：商品详情、用户信息等高频访问数据
- 数据库查询结果缓存：复杂查询的结果缓存
- API响应缓存：RESTful API的响应结果缓存
- 计算结果缓存：价格计算、推荐算法等计算密集型操作的结果

### 方案对比

**方案一：本地内存缓存**
- **优点**：
  - 访问速度极快，毫秒级响应
  - 实现简单，无网络开销
- **缺点**：
  - 内存容量有限，无法缓存大量数据
  - 服务器重启后缓存丢失
  - 多服务器部署时缓存不一致
  - 不适合分布式系统

**方案二：数据库缓存**
- **优点**：
  - 数据持久化，重启后不丢失
  - 支持复杂查询
- **缺点**：
  - 性能提升有限，仍有数据库IO开销
  - 无法缓解数据库压力
  - 并发能力有限

**方案三：Redis分布式缓存**
- **优点**：
  - 高性能，支持每秒百万级操作
  - 数据持久化选项（RDB/AOF）
  - 支持分布式部署，缓存一致性好
  - 丰富的数据结构，支持多种缓存场景
  - 内存管理高效，支持过期时间设置
- **缺点**：
  - 需要额外的Redis服务器资源
  - 有网络传输开销
  - 配置和维护复杂度增加

**选择理由**：
考虑到电商网站的高并发需求和数据一致性要求，选择Redis分布式缓存方案。虽然增加了部署复杂度，但带来的性能提升和可靠性是显著的，能够有效应对促销期间的流量峰值，同时确保数据的实时性和一致性。

### 缓存策略

#### 缓存穿透

**问题**：查询不存在的数据，导致每次都查询数据库

**解决方案**：
- 布隆过滤器
- 缓存空结果
- 接口层参数校验

#### 缓存击穿

**问题**：热点key过期，大量请求同时查询数据库

**解决方案**：
- 热点key永不过期
- 定时更新热点key
- 互斥锁

#### 缓存雪崩

**问题**：大量key同时过期，导致数据库压力剧增

**解决方案**：
- 随机过期时间
- 分层缓存
- 限流降级

### 代码实现

```go
package main

import (
	"context"
	"fmt"
	"time"

	"github.com/go-redis/redis/v8"
)

// CacheService 缓存服务
type CacheService struct {
	rdb *redis.Client
}

// NewCacheService 创建缓存服务
func NewCacheService(rdb *redis.Client) *CacheService {
	return &CacheService{rdb: rdb}
}

// GetWithCache 带缓存的查询
func (s *CacheService) GetWithCache(ctx context.Context, key string, expiration time.Duration, queryFunc func() (string, error)) (string, error) {
	// 1. 尝试从缓存获取
	val, err := s.rdb.Get(ctx, key).Result()
	if err == nil {
		fmt.Println("从缓存获取数据")
		return val, nil
	}

	if err != redis.Nil {
		return "", err
	}

	// 2. 缓存未命中，查询数据源
	val, err = queryFunc()
	if err != nil {
		return "", err
	}

	// 3. 写入缓存
	err = s.rdb.Set(ctx, key, val, expiration).Err()
	if err != nil {
		fmt.Printf("写入缓存失败: %v\n", err)
	}

	fmt.Println("从数据库获取数据并缓存")
	return val, nil
}

// SetCache 设置缓存
func (s *CacheService) SetCache(ctx context.Context, key string, value string, expiration time.Duration) error {
	return s.rdb.Set(ctx, key, value, expiration).Err()
}

// DeleteCache 删除缓存
func (s *CacheService) DeleteCache(ctx context.Context, key string) error {
	return s.rdb.Del(ctx, key).Err()
}

// 使用示例
func main() {
	ctx := context.Background()

	// 创建Redis客户端
	rdb := redis.NewClient(&redis.Options{
		Addr: "localhost:6379",
	})

	// 创建缓存服务
	cacheService := NewCacheService(rdb)

	// 模拟数据库查询函数
	queryFunc := func() (string, error) {
		// 模拟数据库查询
		fmt.Println("执行数据库查询...")
		time.Sleep(100 * time.Millisecond)
		return "数据库查询结果", nil
	}

	// 第一次查询（缓存未命中）
	result1, err := cacheService.GetWithCache(ctx, "user:123", 5*time.Minute, queryFunc)
	if err != nil {
		fmt.Printf("查询失败: %v\n", err)
		return
	}
	fmt.Printf("第一次查询结果: %s\n", result1)

	// 第二次查询（缓存命中）
	result2, err := cacheService.GetWithCache(ctx, "user:123", 5*time.Minute, queryFunc)
	if err != nil {
		fmt.Printf("查询失败: %v\n", err)
		return
	}
	fmt.Printf("第二次查询结果: %s\n", result2)
}
```

### 缓存键设计规范

1. **前缀规范**：`业务:模块:id`
2. **长度控制**：避免过长的键名
3. **语义清晰**：键名要能清晰表达业务含义
4. **使用哈希**：对于复杂参数，使用哈希生成键名
5. **统一管理**：集中管理缓存键的定义

### 缓存监控

- 缓存命中率
- 缓存大小
- 缓存过期统计
- 缓存操作耗时

### 最佳实践

1. **合理设置过期时间**：根据数据更新频率设置
2. **缓存预热**：系统启动时加载热点数据
3. **缓存降级**：当缓存服务不可用时的降级策略
4. **缓存一致性**：保证缓存与数据库数据的一致性
5. **监控告警**：设置缓存相关的监控和告警机制
