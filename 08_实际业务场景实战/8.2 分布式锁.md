## 8.2 分布式锁

### 场景故事：秒杀系统的并发挑战

**故事背景**：
某电商平台在举办限时秒杀活动时，发现系统出现了超卖现象。技术团队分析后发现，当多个用户同时抢购同一商品时，由于并发请求导致库存扣减操作不是原子性的，多个请求同时读取到相同的库存数量，都执行了扣减操作，最终导致实际库存为负数。

**业务需求**：
1. 确保库存扣减操作的原子性，防止超卖
2. 支持高并发场景，秒杀活动期间可能有 thousands of requests per second
3. 保证锁的可靠性，即使服务节点崩溃也能自动释放锁
4. 实现公平锁，避免某些请求长时间等待

### 应用场景

- 分布式系统中的资源竞争：多个服务实例同时操作共享资源
- 并发任务调度：确保任务只被一个节点执行
- 数据一致性保证：保证分布式事务的一致性
- 防止重复操作：避免重复处理相同的请求

### 方案对比

**方案一：基于数据库的分布式锁**
- **优点**：
  - 实现简单，利用数据库的事务特性
  - 可靠性高，数据持久化
- **缺点**：
  - 性能差，数据库成为瓶颈
  - 容易产生死锁
  - 并发能力有限

**方案二：基于ZooKeeper的分布式锁**
- **优点**：
  - 可靠性高，基于ZAB协议
  - 支持公平锁
  - 自动释放锁（会话超时）
- **缺点**：
  - 性能相对较低
  - 部署和维护复杂
  - 资源消耗较大

**方案三：基于Redis的分布式锁**
- **优点**：
  - 高性能，支持高并发
  - 实现简单，API友好
  - 支持过期时间，自动释放锁
  - 内存操作，响应速度快
- **缺点**：
  - 在Redis集群模式下需要考虑一致性
  - 可能存在锁竞争的问题
  - 需要处理锁超时的情况

**选择理由**：
考虑到秒杀系统的高并发需求，选择基于Redis的分布式锁方案。Redis的高性能特性能够满足秒杀场景的并发要求，同时通过合理的锁设计（如设置过期时间、使用Lua脚本保证原子性）可以解决可靠性问题。相比其他方案，Redis分布式锁在性能和实现复杂度之间取得了更好的平衡。

### 分布式锁原理

**核心原理**：利用Redis的原子操作来实现分布式锁

**关键特性**：
- 互斥性：同一时刻只有一个客户端能获取锁
- 安全性：锁只能被持有锁的客户端释放
- 可靠性：即使客户端崩溃，锁也能自动释放
- 可用性：只要Redis集群可用，锁服务就可用

### 分布式锁实现方案

#### 基于SET命令的实现

+----------------+                 +----------------+                 +----------------+

#### 基于Lua脚本的实现

**优势**：
- 原子性操作
- 避免竞态条件
- 减少网络往返

### 代码实现

```go
package main

import (
	"context"
	"fmt"
	"time"

	"github.com/go-redis/redis/v8"
)

// DistributedLock 分布式锁
type DistributedLock struct {
	rdb     *redis.Client
	lockKey string
	value   string
	ttl     time.Duration
}

// NewDistributedLock 创建分布式锁
func NewDistributedLock(rdb *redis.Client, lockKey string, value string, ttl time.Duration) *DistributedLock {
	return &DistributedLock{
		rdb:     rdb,
		lockKey: lockKey,
		value:   value,
		ttl:     ttl,
	}
}

// Lock 获取锁
func (l *DistributedLock) Lock(ctx context.Context) (bool, error) {
	// 使用SET NX EX命令获取锁
	result, err := l.rdb.SetNX(ctx, l.lockKey, l.value, l.ttl).Result()
	if err != nil {
		return false, err
	}
	return result, nil
}

// Unlock 释放锁
func (l *DistributedLock) Unlock(ctx context.Context) (bool, error) {
	// 使用Lua脚本确保原子性释放锁
	script := `
		if redis.call('get', KEYS[1]) == ARGV[1] then
			return redis.call('del', KEYS[1])
		else
			return 0
		end
	`

	result, err := l.rdb.Eval(ctx, script, []string{l.lockKey}, l.value).Result()
	if err != nil {
		return false, err
	}

	return result.(int64) == 1, nil
}

// LockService 分布式锁服务
type LockService struct {
	rdb *redis.Client
}

// NewLockService 创建分布式锁服务
func NewLockService(rdb *redis.Client) *LockService {
	return &LockService{rdb: rdb}
}

// GetLock 获取分布式锁
func (s *LockService) GetLock(ctx context.Context, lockKey string, value string, ttl time.Duration) *DistributedLock {
	return NewDistributedLock(s.rdb, lockKey, value, ttl)
}

// 使用示例
func main() {
	ctx := context.Background()

	// 创建Redis客户端
	rdb := redis.NewClient(&redis.Options{
		Addr: "localhost:6379",
	})

	// 创建锁服务
	lockService := NewLockService(rdb)

	// 获取锁
	lockKey := "lock:order:123"
	lockValue := "client:12345"
	lock := lockService.GetLock(ctx, lockKey, lockValue, 10*time.Second)

	// 尝试获取锁
	locked, err := lock.Lock(ctx)
	if err != nil {
		fmt.Printf("获取锁失败: %v\n", err)
		return
	}

	if locked {
		fmt.Println("获取锁成功")

		// 执行业务逻辑
		fmt.Println("执行业务逻辑...")
		time.Sleep(2 * time.Second)

		// 释放锁
		unlocked, err := lock.Unlock(ctx)
		if err != nil {
			fmt.Printf("释放锁失败: %v\n", err)
			return
		}

		if unlocked {
			fmt.Println("释放锁成功")
		} else {
			fmt.Println("释放锁失败: 锁已被其他客户端持有")
		}
	} else {
		fmt.Println("获取锁失败: 锁已被其他客户端持有")
	}
}
```

### 分布式锁的挑战

1. **锁超时问题**：业务执行时间超过锁的TTL
2. **锁竞争问题**：高并发下的锁竞争
3. **锁重入问题**：同一个客户端多次获取锁
4. **锁释放问题**：客户端崩溃导致锁无法释放

### 解决方案

1. **锁续期**：使用后台线程定期续期锁
2. **公平锁**：使用Redis的List数据结构实现公平锁
3. **可重入锁**：记录客户端获取锁的次数
4. **监控机制**：监控锁的状态和使用情况

### 最佳实践

1. **设置合理的TTL**：根据业务执行时间设置
2. **使用唯一的锁值**：确保只有持有锁的客户端能释放锁
3. **处理锁获取失败**：实现重试机制
4. **监控锁的使用**：设置监控和告警
5. **考虑分布式环境**：在Redis集群环境下的一致性
