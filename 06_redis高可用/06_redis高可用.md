# 6. Redis高可用

## 6.1 主从复制

主从复制是Redis实现高可用的基础，它允许将一个Redis服务器的数据复制到多个从服务器。

### 工作原理

主从复制的工作流程如下：

1. 从服务器连接到主服务器，发送SYNC命令
2. 主服务器执行BGSAVE命令生成RDB文件，同时将新的写命令记录到缓冲区
3. 主服务器将RDB文件发送给从服务器
4. 从服务器加载RDB文件到内存
5. 主服务器将缓冲区中的写命令发送给从服务器
6. 从服务器执行这些命令，完成数据同步
7. 后续主服务器的写命令会实时同步给从服务器

#### 主从复制工作流程图

```
+----------------+     SYNC命令    +----------------+     BGSAVE命令   +----------------+
|  从服务器      | --------------> |  主服务器      | --------------> |  生成RDB文件   |
+----------------+                 +----------------+                 +----------------+
     ^                                   |                                   |
     |                                   v                                   v
+----------------+     执行增量同步  +----------------+     发送RDB文件  +----------------+
|  同步完成      | <-------------- |  缓冲区命令    | --------------> |  传输RDB文件   |
+----------------+                 +----------------+                 +----------------+
     ^                                   |                                   |
     |                                   v                                   v
+----------------+     执行写命令    +----------------+     加载RDB文件  +----------------+
|  数据同步      | <-------------- |  增量命令      | <-------------- |  从服务器加载  |
+----------------+                 +----------------+                 +----------------+
```

#### 主从复制架构图

```
                    +----------------+
                    |   客户端请求    |
                    +----------------+
                           |
           +---------------+---------------+
           |                               |
           v                               v
+----------------+                 +----------------+
|   主服务器     |                 |   从服务器1    |
|   (Master)     |                 |   (Slave)      |
+----------------+                 +----------------+
           |                               |
           +---------------+---------------+
                           |
                           v
                 +----------------+
                 |   从服务器2    |
                 |   (Slave)      |
                 +----------------+
                           |
                           v
                 +----------------+
                 |   从服务器N    |
                 |   (Slave)      |
                 +----------------+
```

### 配置

在从服务器的 `redis.conf` 文件中，添加以下配置：

```
replicaof <masterip> <masterport>
masterauth <masterpassword>
```

### 优点

- 提高读取性能：可以将读请求分发到多个从服务器。
- 提高数据安全性：数据存在多个副本。
- 支持故障转移：当主服务器故障时，可以将从服务器提升为主服务器。

### 缺点

- 主服务器故障时需要手动切换到从服务器。
- 数据同步存在延迟，可能导致数据不一致。

## 6.2 哨兵模式

哨兵模式是Redis实现自动故障转移的方案，它由一个或多个哨兵进程组成，用于监控主从服务器的状态。

### 工作原理

哨兵模式的工作流程如下：

1. 哨兵进程定期向主从服务器发送PING命令，检查服务器状态
2. 当主服务器故障时，哨兵会确认主服务器下线
3. 哨兵进行投票选举，选择一个从服务器提升为主服务器
4. 哨兵将其他从服务器重新配置为指向新的主服务器
5. 通知客户端新的主服务器地址

#### 哨兵模式工作流程图

```
+----------------+     PING命令    +----------------+     PING命令    +----------------+
|  哨兵进程1     | --------------> |   主服务器     | <------------ |  哨兵进程2     |
+----------------+                 +----------------+                 +----------------+
     |                                   |                                   |
     |                                   v                                   |
     |                          +----------------+                          |
     |                          |  从服务器1     |                          |
     |                          +----------------+                          |
     |                                   |                                   |
     |                                   v                                   |
     |                          +----------------+                          |
     |                          |  从服务器2     |                          |
     |                          +----------------+                          |
     |                                   |                                   |
     |                                   v                                   |
     |                          +----------------+                          |
     |                          |  从服务器N     |                          |
     |                          +----------------+                          |
     |                                                                   |
     |                          主服务器故障                             |
     |                                                                   |
     v                                                                   v
+----------------+     投票选举    +----------------+     选举新主    +----------------+
|  确认下线      | --------------> |  哨兵投票      | --------------> |  提升从服务器   |
+----------------+                 +----------------+                 +----------------+
     |                                                                   |
     v                                                                   v
+----------------+     通知客户端  +----------------+     重新配置    +----------------+
|  故障转移完成  | --------------> |  客户端更新    | --------------> |  从服务器重连   |
+----------------+                 +----------------+                 +----------------+
```

#### 哨兵模式架构图

```
+----------------+                 +----------------+                 +----------------+
|  哨兵进程1     |                 |  哨兵进程2     |                 |  哨兵进程N     |
+----------------+                 +----------------+                 +----------------+
        |                                   |                                   |
        +-------------------+---------------+-------------------+               |
                            |                                   |               |
                            v                                   v               v
                    +----------------+                 +----------------+----------------+
                    |   主服务器     |                 |   从服务器1     |   从服务器N     |
                    |   (Master)     |                 |   (Slave)       |   (Slave)       |
                    +----------------+                 +----------------+----------------+
                            |                                   |               |
                            +-------------------+---------------+               |
                                                |                               |
                                                v                               v
                                        +----------------+             +----------------+
                                        |   客户端1      |             |   客户端N      |
                                        +----------------+             +----------------+
```

### 配置

创建一个 `sentinel.conf` 文件，添加以下配置：

```
sentinel monitor <master-name> <masterip> <masterport> <quorum>
sentinel auth-pass <master-name> <password>
sentinel down-after-milliseconds <master-name> <milliseconds>
sentinel failover-timeout <master-name> <milliseconds>
```

### 优点

- 自动故障转移：当主服务器故障时，哨兵会自动将一个从服务器提升为主服务器。
- 监控功能：哨兵会监控主从服务器的状态，及时发现故障。

### 缺点

- 需要部署多个哨兵进程，增加了系统复杂度。
- 故障转移过程中可能会有短暂的服务中断。

## 6.3 集群模式

Redis集群是Redis 3.0引入的分布式解决方案，它将数据分布在多个节点上，实现了数据的自动分片和高可用。

### 工作原理

Redis集群使用哈希槽（hash slot）机制来实现数据分片：

1. 集群共有16384个哈希槽（0-16383）
2. 每个节点负责处理一部分哈希槽
3. 当客户端请求某个键时，Redis会计算键的哈希值，然后对16384取模，确定该键属于哪个哈希槽
4. 客户端将请求发送到负责该哈希槽的节点
5. 每个主节点可以有多个从节点，用于故障转移

#### 哈希槽分布示意图

```
+----------------+----------------+----------------+----------------+
|   哈希槽0-4095 |  哈希槽4096-8191 | 哈希槽8192-12287 | 哈希槽12288-16383 |
+----------------+----------------+----------------+----------------+
|   节点A        |   节点B         |   节点C         |   节点D         |
+----------------+----------------+----------------+----------------+
```

#### 集群模式架构图

```
+----------------+                 +----------------+
|   客户端1      |                 |   客户端N      |
+----------------+                 +----------------+
        |                                   |
        +-------------------+---------------+-------------------+
                            |                                   |
                            v                                   v
                    +----------------+                 +----------------+
                    |   节点A        |                 |   节点B        |
                    |   槽:0-4095    |                 |   槽:4096-8191 |
                    +----------------+                 +----------------+
                            |                                   |
                            +---------------+-------------------+
                                            |
                                            v
                                    +----------------+
                                    |   节点C        |
                                    |   槽:8192-12287|
                                    +----------------+
                                            |
                                            v
                                    +----------------+
                                    |   节点D        |
                                    |   槽:12288-16383|
                                    +----------------+
```

#### 集群模式故障转移示意图

```
正常状态：
+----------------+                 +----------------+
|   节点A        |                 |   节点B        |
|   主节点       |                 |   主节点       |
+----------------+                 +----------------+
        |                                   |
        v                                   v
+----------------+                 +----------------+
|   从节点A1     |                 |   从节点B1     |
+----------------+                 +----------------+

节点A故障后：
+----------------+                 +----------------+
|   节点A        |                 |   节点B        |
|   主节点(故障) |                 |   主节点       |
+----------------+                 +----------------+
        |                                   |
        v                                   v
+----------------+                 +----------------+
|   从节点A1     |                 |   从节点B1     |
|   提升为主节点 |                 +----------------+
+----------------+
```

### 配置

创建一个 `redis.conf` 文件，添加以下配置：

```
cluster-enabled yes
cluster-config-file nodes.conf
cluster-node-timeout 15000
```

### 优点

- 水平扩展：可以通过添加节点来扩展Redis的存储容量和处理能力。
- 自动分片：数据会自动分布在多个节点上。
- 高可用：当部分节点故障时，集群仍然可以正常工作。

### 缺点

- 集群配置较为复杂，需要多个节点。
- 不支持跨节点的多键操作。
- 客户端需要支持集群模式。
